# ROADMAP MESTRE — CANVAS KIT IA

IMPORTANTE:
Este arquivo é o plano completo do módulo Canvas KIT IA.
Não execute tudo de uma vez.
Quando o usuário pedir "executar etapa X", execute somente aquela etapa.
Nunca remova etapas futuras.
Nunca reescreva a arquitetura sem necessidade.
Antes de editar, leia o estado atual do projeto.
Ao finalizar uma etapa, marcar como concluída no checklist deste arquivo.

==================================================
0. VISÃO GERAL
==================================================

O Canvas KIT IA é um editor modular local-first integrado ao Electron da KIT IA.

Objetivo:
Criar um motor híbrido de criação visual:
- Automação por templates
- Edição manual criativa
- Integração futura com IA local
- Exportação para redes sociais
- Suporte futuro a áudio, vídeo e timeline

O sistema deve rodar offline sempre que possível, usando GPU local para IA e processamento gráfico.

==================================================
1. ARQUITETURA DE ARQUIVOS
==================================================

O sistema usa dois tipos principais de arquivo:

1.1 Arquivo .kit

Arquivo global de marca.

Contém:
- Identidade visual
- Logos PNG/SVG
- Paleta de cores HEX
- Fontes padrão
- Voz da marca
- Configuração XTTS
- Assets globais
- Molduras
- Marcas d'água
- Elementos recorrentes

1.2 Arquivo .kia

Arquivo local de projeto.

Contém:
- Referência ao .kit pai
- Estado serializado do Fabric.js
- Posições
- Escalas
- Camadas
- Metadados de IA
- Prompts usados
- Histórico de gerações
- Máscaras de inpainting
- Timeline
- Slides
- Áudios
- Vídeos
- Duração dos elementos

Regra:
O .kia herda dados do .kit, mas pode sobrescrever configurações locais quando necessário.

[x] ETAPA 14 — Stable Diffusion Worker Base

Objetivo:
Criar um serviço local próprio da KIT para geração e edição básica de imagens via Stable Diffusion, sem uso de ComfyUI, Forge ou WebUI.

Inclui:
- Criar worker Python local usando Diffusers
- Criar API HTTP local para o worker
- Criar endpoints mínimos:
  - /health
  - /models
  - /txt2img
  - /img2img
  - /inpaint
- Backend da KIT deve consumir essa API local
- Canvas deve conseguir solicitar geração usando o artboard ativo como contexto
- Salvar outputs em pasta dedicada, como /output/sd
- Registrar no .kia:
  - prompt positivo
  - prompt negativo
  - checkpoint usado
  - LoRA usado
  - seed
  - modo usado
  - width
  - height
  - sampler/scheduler
  - arquivo de saída gerado

Arquitetura suportada:
- Checkpoints SD 1.5
- Checkpoints SDXL
- Modelos locais .safetensors e .ckpt
- Detecção automática do tipo do modelo quando possível
- Permitir seleção manual do tipo do modelo se a detecção falhar

Regras de carregamento:
- Carregar modelo sob demanda (lazy load)
- Não manter dois checkpoints pesados carregados ao mesmo tempo
- Antes de carregar novo checkpoint:
  - descarregar o anterior
  - limpar cache CUDA
- Não usar refiner nesta fase
- Não implementar ControlNet nesta fase
- Não implementar upscale/highres fix nesta fase

Execução em GPU:
- Priorizar sempre CUDA na RTX disponível
- Usar fp16 quando suportado
- Usar xformers se disponível
- Aplicar otimizações de memória compatíveis com Diffusers
- Limpar VRAM após geração quando necessário

Regra de proporção com o Canvas:
- O worker não deve gerar usando resolução fixa sem contexto
- Antes de gerar, o backend deve consultar o Canvas ativo e obter:
  - preset atual do artboard
  - largura do artboard
  - altura do artboard
  - proporção atual
- A geração deve herdar a proporção do artboard ativo
- O tamanho final exportado continua sendo responsabilidade do Canvas
- O tamanho de geração é responsabilidade do SD Worker

Presets por proporção:
- instagram_post → 1:1
- instagram_story → 9:16
- instagram_reels → 9:16
- facebook_post → 1:1 ou 4:5
- youtube_thumb → 16:9
- custom → proporção derivada do artboard ativo

Resoluções padrão por arquitetura:
SD 1.5:
- 1:1 → 512x512
- 4:5 → 512x640
- 9:16 → 512x912
- 16:9 → 768x432

SDXL:
- 1:1 → 1024x1024
- 4:5 → 1024x1280
- 9:16 → 1024x1792
- 16:9 → 1344x768

Fallbacks:
- SDXL deve ter fallback para resolução menor se faltar VRAM
- Width e Height podem ser alterados manualmente, mas por padrão devem seguir o artboard ativo
- Após gerar, a imagem deve ser inserida no Canvas e adaptada ao artboard
- Inicialmente usar modo de encaixe padrão "cover"

Controles obrigatórios do painel Stable Diffusion:
Campos principais:
- Prompt positivo
- Prompt negativo

Seleção de modelo:
- Checkpoint
  - Lista expansível lateral
  - Exibir nome do modelo
  - Se possível, exibir miniatura/preview
  - Ao selecionar, fechar menu automaticamente

- LoRA
  - Lista expansível lateral
  - Exibir nome do LoRA
  - Se possível, exibir miniatura/preview
  - Permitir selecionar e remover LoRA
  - Ao selecionar, fechar menu automaticamente

Configurações de geração:
- Sampling method
- Schedule type
- Sampling steps
- Width
- Height
- CFG Scale
- Seed
  - Valor padrão: -1
  - -1 significa seed aleatória

Modos:
- txt2img
- img2img
- Sketch
- Inpaint
- Inpaint sketch
- Inpaint upload
- Batch

Controles específicos de img2img/inpaint:
- CFG Scale
- Denoising strength

Batch:
- Batch count fixo em 1 na Fase 1
- Batch size fixo em 1 na Fase 1

Regra:
- Na Fase 1, gerar apenas uma imagem por vez
- O objetivo é reduzir consumo de VRAM e evitar travamentos

Modo avançado:
- Exibir os nomes técnicos reais dos schedulers do Diffusers
- Exemplos:
  - DPMSolverMultistepScheduler
  - EulerDiscreteScheduler
  - EulerAncestralDiscreteScheduler
  - DDIMScheduler
  - LMSDiscreteScheduler

Critério de conclusão:
A KIT consegue selecionar um checkpoint local, usar a proporção do artboard ativo, gerar imagem via worker próprio e inserir o resultado no Canvas.

Não incluir ainda:
- vídeo com IA
- geração de áudio por diffusion
- ControlNet
- upscaling avançado
- highres fix
- refiner
- batch maior que 1

[x] ETAPA 20 — Timeline de Mídia estilo edição de vídeo
Objetivo

Implementar a base funcional da timeline da KIT como um sistema de edição temporal estilo editor de vídeo, permitindo organizar, posicionar e manipular mídias ao longo do tempo do projeto.

A timeline é a referência principal de tempo do projeto.

Os slides não são páginas rígidas.
Eles existem apenas como agrupamento opcional de organização, navegação e thumbnail.

Conceito principal
A timeline é global e representa o tempo real do projeto inteiro.
O editor deve funcionar com lógica de edição de vídeo, não com lógica presa a páginas estáticas.
Slides são opcionais como estrutura de organização.
Um projeto pode ter:
vários slides sequenciais
ou apenas 1 slide cobrindo quase todo o projeto
Itens podem ultrapassar o tempo visual do slide.
Como regra de organização, o item pertence ao slide onde ele começa.
Se um slide for apagado, todos os itens filhos dele também devem ser apagados.
Inclui
Estrutura da Timeline

Criar painel de timeline no editor com comportamento de edição temporal horizontal.

A timeline deve ter:

régua horizontal de tempo
cursor/ponteiro de reprodução
trilhas empilhadas
blocos visuais representando os itens
seleção visual de item
movimentação horizontal no tempo
ajuste de duração nas extremidades
Estrutura mínima dos itens da timeline

Cada item da timeline deve possuir pelo menos:

id
type
startTime
duration
track
source
label

Campos recomendados para já deixar preparado:

slideId
visible
muted
volume
speed
locked
Tipos iniciais de item
imagem
vídeo
música
texto
áudio
Regras dos Slides
Slide é um agrupador temporal opcional.
Slide não deve travar a liberdade da timeline.
Todo item pode existir normalmente na timeline mesmo ultrapassando o intervalo visual do slide.
O item pertence ao slide em que inicia.
O slide serve para:
organização
navegação
thumbnail
referência inicial de inserção
Ao apagar um slide, apagar também todos os itens filhos vinculados a ele.
Regra importante
O item pode ultrapassar o fim do slide.
Mesmo ultrapassando, ele continua pertencendo ao slide onde começou.
O slide não deve cortar, travar ou limitar artificialmente o item.
Exemplo
Slide 1: 0s até 5s
Um áudio começa em 4s e termina em 12s
Esse áudio pertence ao Slide 1
Ele continua existindo normalmente além do tempo visual do slide
Inserção padrão de itens

Ao inserir um novo item:

ele entra na timeline do projeto
entra por padrão no slide ativo, se houver
entra por padrão no início temporal do slide ativo
depois disso pode ser movido livremente
não deve ficar preso ao intervalo do slide
Controles visuais iniciais

Usar os ícones da pasta:

F:\AI\Ai_kit\kit-app\renderer\assets\icones\timeline

Mapeamento inicial sugerido
filme.svg → vídeo

tacometro.svg → velocidade
tesouras.svg → cortar

um-passo-a-frente.svg → próximo
um-passo-atras.svg → anterior
Regra visual dos ícones
Os ícones da timeline devem ser exibidos em tamanho pequeno, mesmo que o arquivo SVG original tenha alta resolução.
O objetivo é aparência de software de edição, não botões grandes de app mobile.
Priorizar leitura limpa, compacta e profissional.
Interações mínimas da timeline

Implementar no mínimo:

selecionar item
mover item no tempo
alterar duração
alternar visibilidade
cortar item
ajustar volume
ajustar velocidade
inserir item em trilha
Regras da timeline
A timeline é do projeto inteiro
Não vincular áudio a slide específico como limitação estrutural
Deve permitir múltiplos áudios sobrepostos
Deve permitir múltiplos itens em trilhas diferentes
Deve funcionar como edição de vídeo com liberdade temporal
Deve permitir, por exemplo
música de fundo contínua
trechos curtos de narração
efeitos separados em outras trilhas
vídeo em uma trilha
texto temporário em outra
imagem atravessando parte de um slide e entrando no próximo visualmente
Bridge para a KIT IA

Expor comandos para a KIT operar a timeline via chat/voz:

adicionar mídia na timeline
mover item
cortar item
ocultar / exibir item
alterar duração
ajustar volume
ajustar velocidade
Regra da bridge

A KIT deve operar a timeline como sistema de diagramação temporal real, com permissão para organizar os itens no tempo do projeto.

Não incluir nesta etapa
geração XTTS
watcher de pasta do XTTS
conversão automática de legenda em áudio
conversão automática de áudio em legenda
waveform
mixagem avançada
exportação MP4
transições avançadas
automações de edição por IA
Não fazer
não tratar slide como página rígida
não limitar item ao intervalo visual do slide
não prender áudio semanticamente ao slide
não impedir sobreposição de mídias
não transformar a timeline em sequência fechada por página
não usar ícones grandes na UI da timeline
Critério de conclusão

A etapa estará concluída quando:

a timeline existir e funcionar como base real de edição temporal estilo vídeo
itens puderem ser adicionados e manipulados visualmente
itens puderem ser movidos, redimensionados e sobrepostos em trilhas
slides funcionarem apenas como agrupamento opcional
itens puderem ultrapassar o tempo visual do slide
cada item pertencer ao slide onde inicia
ao apagar um slide, todos os itens filhos dele forem apagados
a KIT IA conseguir inserir e editar itens da timeline pela bridge de comandos
áudios puderem coexistir sobrepostos na timeline do projeto inteiro
os ícones da timeline forem exibidos pequenos, com aparência de editor profissional--------------------------------------------------
